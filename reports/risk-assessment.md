# Risk Assessment Table

Ratings reflect the environment as tested on 2026-07-11 (`security-tests/run_all_tests.sh`,
8/8 passed — see `reports/testing-matrix.md`). Likelihood is rated *as
deployed in this project*, i.e. with the listed control in place; the
"Recommended Improvement" column is what would further reduce Likelihood or
Impact before any production use, per `reports/conclusion-recommendations.md`.

| # | Risk | Impact | Likelihood | Existing Control | Recommended Improvement |
|---|------|--------|------------|-------------------|--------------------------|
| 1 | Anonymous/unauthenticated read of a bucket | High (data disclosure) | Low — denied by default bucket policy; confirmed via `test_01` (anonymous `curl` → 403, bad credentials → auth error) | No public/anonymous bucket policy; every read requires a signed request | Add bucket policy monitoring/alerting on repeated 403s (possible enumeration/credential-stuffing attempt) |
| 2 | Privilege escalation / cross-department access | High | Low — RBAC enforced; confirmed via `test_02` (hr-manager → finance-documents denied, finance-analyst write attempt denied, guest-contractor denied everywhere) | Three scoped IAM policies (`hr-readwrite`, `finance-readonly`, `auditor-readonly`), one identity per department, no shared credentials | Periodic access review of policy assignments; move to short-lived STS credentials instead of long-lived static keys |
| 3 | Data interception in transit | High | Low — TLS-only listener; confirmed via `test_03` (TLSv1.3 handshake verified against our CA, plaintext HTTP request rejected) | OpenSSL-issued cert, MinIO configured TLS-only on API + console ports | Replace the self-signed CA with a CA-signed certificate (internal PKI or public CA) before any production/external use |
| 4 | Data theft from disk / storage media (e.g. stolen drive, unauthorized host access) | Medium–High | Low — SSE-S3 default encryption; confirmed via `test_04` (on-disk bytes are ciphertext, not plaintext JSON) | Default bucket encryption (AES-256) on all four buckets | Move from the static `MINIO_KMS_SECRET_KEY` to an external KMS (Vault, AWS KMS, or MinIO KES) with key rotation and separated access control |
| 5 | Undetected data corruption/tampering | Medium | Low — confirmed via `test_05`: MinIO's internal bitrot/checksum protection independently rejected a deliberately corrupted object, and the external SHA-256 manifest caught it as a second layer | Bucket versioning + MinIO bitrot protection + external SHA-256 manifest | Automate periodic integrity sweeps against the manifest and alert on mismatch, rather than relying on ad-hoc verification |
| 6 | Actions not attributable / no forensic trail | Medium | Low — confirmed via `test_06`: both allowed and denied actions were captured with accessKey, API name, and status code, and queryable | Audit webhook receiver persisting every S3 call (allow + deny) to an append-only log | Ship logs to a durable, tamper-evident SIEM (ELK/Loki) with a defined retention policy instead of a local JSON-lines file |
| 7 | Malware-laden document uploaded to storage | Medium–High | Medium — ClamAV correctly identified the EICAR test file and cleared legitimate documents in `test_07`, but scanning is on-demand, not enforced on every upload | ClamAV scan available and verified effective against known signatures | Wire ClamAV into MinIO bucket-event notifications so every `PutObject` is scanned before being considered available, with a quarantine bucket for hits |
| 8 | Data loss from accidental deletion | Medium | Low — confirmed via `test_08`: deleted object was unreadable, then fully recovered via both versioning and the mirrored backup bucket | Bucket versioning + `mc mirror` backup to a separate `backups` bucket | Add an off-site/off-host backup target and run the mirror job on a monitored schedule rather than on demand |
| 9 | Root/admin credential compromise | Critical | Low — root credentials are only used for initial setup/admin tasks, not day-to-day department access, and are kept in a gitignored `.env` | Root creds isolated from department users; not embedded in application code | Use per-user credentials exclusively for daily operations, disable/rotate the root key, enable MFA where the deployment target supports it |
| 10 | Self-signed CA not trusted by clients | Low | High (by design in this lab — every client uses `--insecure` or manually trusts `certs/ca.crt`) | Root CA + server cert issued in-house with OpenSSL | Replace with a CA-signed certificate before any deployment outside the lab environment |
| 11 | Insider risk from the root/admin account bypassing RBAC | High | Low–Medium — a single compromised or misused root credential bypasses all bucket-level policy by design | Root account not used for routine access; separate department identities exist for normal operations | Restrict root credential use to break-glass scenarios only, with logging/alerting on any root-account activity |
| 12 | Audit log tampering at the host level | Medium | Low — not directly tested; the audit-webhook receiver and its log files run on the same Docker host as MinIO | Append-only log file, but no write-once storage or off-host replication | Ship logs off-host in near-real-time (SIEM) so a compromised host cannot retroactively erase its own audit trail |
| 13 | Lack of network segmentation | Medium | Low — all services share a single Docker bridge network (`storage-net`) with no further segmentation | Services isolated from the host network except explicitly published ports | In a multi-tenant/production deployment, segment storage, logging, and scanning services onto separate network zones with least-privilege firewall rules |

**Summary:** all ten scenario-relevant risks (rows 1–10) were exercised by
the eight security tests and found to be mitigated as designed. Rows 11–13
are risks identified through architectural review during the build, not
directly exercised by a scripted test, and are flagged as residual risk in
`reports/conclusion-recommendations.md`.
